# 游戏配额管理与全局配置 — 测试用例

**依据**：[游戏配额管理与全局配置需求文档](./游戏配额管理与全局配置需求文档.md)  
**对照原型**：Publisher-demo（`GameEnvDetail.tsx`、`ResourceConfiguration.tsx`、管理台 `admin/page.tsx`、`GameManagement.tsx` 等）

本文档中「需求预期」与「原型现状」不一致处已标注，便于联调与验收对齐。

---

## 一、游戏配额管理 — 环境与部署类型（P0）

| 编号 | 场景 | 前置条件 | 步骤 | 预期结果 |
|------|------|----------|------|----------|
| GQ-01 | test / prod 独立配置 | 进入游戏详情 → 「资源限额配置」 | 分别编辑测试环境与正式环境的配额并保存 | 两组数据互不影响；保存后切换 Tab/环境展示的值为各自保存结果（符合「两套配额独立保存」）。 |
| GQ-02 | 容器部署仅展示容器字段 | `deployType === 'container'` 的游戏（如原型 `testgame`） | 进入限额编辑，查看「计算资源」卡片 | 展示：单个应用 CPU、Pod 副本上限、应用数、定时任务数、镜像数等；**不展示**虚机 CPU/内存/磁盘字段。 |
| GQ-03 | 虚机部署仅展示虚机字段 | `deployType === 'vm'` 的游戏（如 `gamedemo`） | 同上 | 展示：虚机 CPU、内存、磁盘总容量；**不展示**容器侧 Pod/应用/Cron/镜像上限。 |
| GQ-04 | 容器字段边界 — 需求口径 | 容器游戏，限额编辑 | 按需求表校验：`maxPodReplicasPerPod` 1–30、`maxAppsPerGame` 1–50、`maxCronJobsPerGame` 1–20、`maxImagesPerGame` 1–50；`cpuCores` 按文档「约束」列（文档为「110」，**联调需确认实际区间**） | **需求**：提交在合法区间内；非法提示明确。**原型现状**：`cpuCores` 等为 1–64、副本/应用等上限与文档不同，可记验收差异或缺陷。 |
| GQ-05 | 虚机字段边界 | 虚机游戏，限额编辑 | 输入：`vmCpuCores` 1–128、`memoryGB` 1–2048、`vmDiskTotalSizeGB` 1–65536 | **需求**：与表一致。**原型**：与上述范围一致则通过；默认值与文档 test/prod 默认值差异可单列用例。 |
| GQ-06 | 不得低于已使用量 | 限额编辑页（原型带 `computeUsed` 校验） | 将任一带「不能小于已使用」的字段调到小于 mock 已使用值并失焦/提交 | 表单项报错，文案含「不能小于已使用值」；确认弹窗不出现或最终不落地。 |
| GQ-07 | 全表单校验通过才可提交 | 限额编辑 | 漏填必填项点「确认」 | 表单提示错误；不弹出二次确认或确认后不更新。 |
| GQ-08 | 保存二次确认 | 限额合法 | 填写合法值 → 「确认」 | 弹出配额保存确认弹窗 → 弹窗内再次确认后写入状态并提示成功。 |
| GQ-09 | 已用/未用展示（接口联调） | 后端提供已用量 | 打开限额页 | 展示实时已用、剩余或进度（需求 3.1.4）；原型若仅 Mock，待接口就绪后执行。 |

## 二、健康检查规则 — 云原生（P0）

| 编号 | 场景 | 步骤 | 预期结果 |
|------|------|------|----------|
| HC-01 | 默认关闭（需求） | 新游戏或无配置数据 | **需求**：默认「不开启」。**原型**：mock 多为已开启，需对齐产品或标「默认策略」差异。 |
| HC-02 | 开关只读/编辑 | 健康检查 Tab | 只读态展示是否开启、间隔、阈值列表；点「编辑」进入表单。 |
| HC-03 | 检查间隔默认 60s（需求） | 新建或为默认数据 | **需求**：默认 60。**原型**：mock 为 60 则通过。 |
| HC-04 | 间隔输入校验 | 编辑态将间隔设为 0 或空 | 提示非法；**需求**写间隔无上限，**原型** `min: 1`，无上限口径需产品与研发统一后再验收。 |
| HC-05 | 应用流量阈值多条 | 添加多行应用 + 阈值 | 多条展示为「流量低于平均值 X%」；保存后与列表一致。 |
| HC-06 | 与间隔为「或」关系（业务逻辑） | 配置评审/集成测试 | 后端或监控策略：**满足间隔或任一应用流量条件即触发**（需接口说明）；UI 配置项齐备。 |
| HC-07 | 保存二次确认 | 合法填写 → 确认 | 弹出「确认保存健康检查规则」，确认后成功提示且仅当前环境生效。 |

## 三、自动开服是否可回退（P1）

| 编号 | 场景 | 步骤 | 预期结果 |
|------|------|------|----------|
| AR-01 | 配置读写 | 「自动开服配置」→ 查看/编辑 | 展示「自动开服回退」开关；只读「已开启/未开启」，编辑用 Switch。 |
| AR-02 | 默认值可回退（需求） | 新环境 | **需求**：默认可回退。**原型**：演示数据可能与需求不一致，需产品确认。 |
| AR-03 | 保存确认 | 修改步骤与回退开关 → 确认 | 弹出二次确认，文案含对开服流程/回退的影响说明。 |
| AR-04 | 开服页「回退开服」联动（端到端） | 关闭 vs 开启回退的游戏 | **需求**：可回退时开服页面展示「回退开服」按钮。**原型**：`ContainerApplication` / `AutoServerModule` 含静态按钮，与 `GameEnvDetail` 配置**未见代码绑定**——建议：**回退关闭时按钮隐藏或禁用**；标记为端到端验收项。 |

## 四、MSE 实例类型与网关配置（P1）

| 编号 | 场景 | 步骤 | 预期结果 |

## 五、全局监控配置（P0）

| 编号 | 场景 | 步骤 | 预期结果 |
|------|------|------|----------|
| GM-01 | 与资源全局配置命名区分 | 管理台「游戏资源配置」vs 游戏详情 Grafana/CDN | 模块命名、入口不混淆。 |
| GM-02 | 待配置 / 已配置 | Grafana、CDN 未完成 | 显示「待配置」与「完成」操作入口。 |
| GM-03 | 完成前确认框 | 点击「完成」 | 弹出确认框「确认完成配置？」；取消则状态不变。 |
| GM-04 | 确认后状态刷新 | 确认 | 对应项变为已配置；提示成功；页面状态立即更新。 |
| GM-05 | 操作不可逆（需求风险） | 完成后 | **无官方回滚**；不提供将已完成项改回未完成的默认路径（持久化后刷新仍保持已配置）。 |

## 六、资源全局配置 — ResourceConfiguration（P0）

| 编号 | 场景 | 步骤 | 预期结果 |
|------|------|------|----------|
| RG-01 | 入口与标题 | 管理台侧栏「游戏资源配置」 | 展示「资源全局配置」标题；按测试环境、正式环境分块。**文档**：stg-admin / pro-admin / dev-admin 与角色的映射以后端权限为准。 |
| RG-02 | test / prod 独立编辑 | 各环境「编辑」「取消」「确认」 | 两环境配置互不影响；成功提示区分环境。 |
| RG-03 | storageInstancesLimit 边界 | 输入边界值 | **需求**：1–100。**原型**：表单允许 0–100，与需求不符处需对齐其一。 |
| RG-04 | 复制生产数据 / 备份游戏 | 编辑正式环境段落 | **需求**：`storageBackupGames` 与 stg 描述相关。**原型**：备份游戏多选在 prod 段落且必选——环境与必填规则需产品与文档对齐后再断言。 |
| RG-05 | vmAllowedSpecs 多选 | 选择多个规格 | 保存后以「、」展示；可选集与文档/阿里云数据源一致（由接口或配置拉全量后截取）。 |
| RG-06 | MySQL / Redis / Mongo / ZK | 多选与子段区间 | Redis 分片数量、Mongo 各计数区间等：**需求见文档表格**。**原型**：Mongo 部分区间为 0–20，ZK 节点 1–9；文档 ZK 为 1–3——差异记入验收对照表。 |
| RG-07 | 区间逆序校验 | Redis/Mongo 任一段 max&lt;min 后提交 | 提示「结束值需大于等于起始值」，不写入。 |
| RG-08 | 保存即时生效（需求） | 保存成功后 | CP 可选规格受限；前端展示为最新值（依赖后端契约）。 |

## 附录：需求 vs 原型校验对照（快速差异清单）

| 项 | 需求文档 | Publisher-demo 原型（实现侧） |
|----|----------|--------------------------------|
